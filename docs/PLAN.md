# 협업 계획 — 컴플라이언스 서류 자동 검증 AI Copilot

> KB국민은행 제8회 AI Challenge / 마감 ~2026-08-03 / 2인 팀 + 각자의 LLM 세션

## 1. 목표와 포지셔닝

행원이 제출받는 서류를 자동 파싱하고 금융 규제·법령과 대조하여
**누락 항목·위험 조항을 적출하고 근거 조문을 하이라이트**하는 B2E Copilot.

- 검증 대상 업무는 **1개로 고정** (1~3일차에 확정: 기업대출 신청 서류 vs 금융상품 판매 서류 중 택1)
- 차별화 문장: "7회 입상작이 규정↔규정 모니터링이라면, 우리는 제출 서류↔규정 검증"
- 심사 어필 3요소: ①근거 조문 하이라이트(XAI) ②규정 개정 시 자동 재검증 ③정량 지표(검증 시간 단축률, 누락 적발률)
- 만들지 않는 것: 대화형 법령 해석 챗봇(작년 입상작 기능), 다중 업무 지원 — "향후 로드맵"으로만 기재

## 2. 역할 분담 (모듈 소유권)

| Track | 모듈 | 담당 | 내용 |
|---|---|---|---|
| A | `src/ingest/` | (미정 — 상의 후 기입) | 국가법령정보 API 연동, 규정 수집·청킹, RAG 인덱스 |
| A | `src/verify/` | (Track A 담당) | 서류 항목 ↔ 규정 대조, 누락/위험 판정, 근거 조문 추출 |
| B | `src/parser/` | (미정 — 상의 후 기입) | 서류 업로드 → 파싱/OCR → 구조화(ParsedDocument) |
| B | `app/` | (Track B 담당) | Streamlit 데모 UI, 리포트 시각화, 발표 데모 |
| 공용 | `src/common/`, `docs/` | 공동 (변경 시 상의 + 단독 PR) | 인터페이스 스키마, 문서 |

> 담당을 정하면 이 표의 "(미정)"을 이름으로 바꾸는 PR을 올릴 것.
> 상대 모듈 수정이 필요하면 직접 고치지 말고 Issue로 요청 — 충돌 방지의 핵심.

## 3. 브랜치 전략 / 작업 흐름

```
main (보호: 직접 푸시 금지, PR로만 머지)
 ├─ feat/tommy/…    ← 사용자(tommy0111) + 사용자의 LLM 세션
 └─ feat/m0282/…    ← 친구(M0282) + 친구의 LLM 세션
```

1. GitHub Issue 생성(작업 클레임) → `feat/<이름>/<작업>` 브랜치 생성
2. 작업 → 커밋(`<type>: <요약>`) → 푸시 → PR
3. **상대방이 GitHub에서 확인 후 머지** (셀프 머지는 급할 때만, 사후 공유)
4. 머지되면 양쪽 모두 `git pull origin main` 후 다음 작업
- PR은 하루 단위 이하로 작게 — 오래 끌수록 충돌 확률이 커진다
- 같은 파일을 동시에 만지는 상황 자체를 만들지 않는 것이 목표 (모듈 소유권)

## 4. 인터페이스 계약

모듈 간 데이터 구조는 **`src/common/schemas.py`가 유일한 진실**이다.

```
[parser]  ParsedDocument(문서ID, 유형, 추출 필드들, 원문)
    ↓
[verify]  VerificationReport(문서ID, 체크 목록[규칙, 판정, 근거 조문, 제안], 요약)
    ↓
[app]     리포트 시각화 (누락=빨강, 위험=주황, 통과=초록 + 조문 클릭 시 원문)
```

- 스키마 변경 = 상대 모듈 파손. **변경 전 상의, 변경은 단독 PR**
- 이 계약 덕분에 Track A와 B는 서로를 기다리지 않고 병렬 개발 가능 (상대 모듈은 mock으로 대체)
