## 반드시 가져올 것

### 1. 근거 텍스트 존재 검증 (환각 방지) — `src/verify`에 반영
LLM이 "이 문장이 근거"라고 내놓은 evidence가 **원본 문서에 실제로 존재하는지 재확인**하고,
없으면 그 판정을 버린다. (친구 코드 `rule_engine._evidence_exists`: 공백 제거 후 부분일치 + 앞 20자 재확인)
- 우리처럼 근거 조문·근거 문장을 전면에 내세우는 제품에서는 **필수 안전장치**.
- 서류 쪽 evidence는 `pdf_loader.locate()`로 좌표까지 확인되면 존재가 자동 보장됨 → 이미 절반 확보.

### 2. 패키지 단위 교차 검증 — 우리 주제에서 오히려 더 강력
여러 문서 사이의 값 불일치를 잡는 아이디어. 친구는 회사명·사업자번호 불일치를 검출.
- 우리 적용: **적합성 진단표(투자성향) ↔ 상품설명서(위험등급)** 대조 →
  "안정형 고객에게 고위험 상품" 같은 금소법 17조(적합성원칙) 위반을 교차 검증으로 적발.
- 단일 문서만 봐선 안 나오고 두 문서를 대조해야만 나오는 결과라 기술적 설득력이 큼.

### 3. 하이브리드 설계 원칙 명문화
"LLM = 문서 분류·정보 추출·유사 조항 탐색·설명/수정안 생성 /
 규칙(코드) = 필수서류 누락·값 불일치·최종 상태 계산" — 우리 PLAN과 동일 철학.
- 형식 검증은 규칙, 의미 판단만 LLM → LLM 호출량↓(비용↓) + 발표에서 엔지니어링 성숙도 어필.
